# Plan: lms-mvp

## Goal

Build the MVP of an independent, production-ready, publicly self-service **multi-tenant SaaS** online learning platform (Tasks 1-6 of `plan.md`'s roadmap): any customer can sign up and create their own organization from launch, a teacher in that organization can author and publish a course, a learner can register, pay, learn, submit work, and earn a certificate — with every organization's data fully isolated from every other's. Everything beyond Tasks 1-6 (communities, AI tools, SCORM, podcasts, the full admin console with feature flags/quotas/CLI/backups, mobile, full hardening) is explicitly out of scope for this plan — Task 6 does include a minimal read-only admin dashboard (org-scoped + platform-owner cross-org view), distinct from that later full admin console.

## Approach

Go (Gin) backend that both server-renders HTML (HTMX + Tailwind) and exposes a versioned JSON API from day one, backed by Supabase Postgres with Row-Level Security as the tenant-isolation boundary, Supabase Auth as the identity source of truth, bunny.net for video, Supabase Storage for other files, Redis-backed background jobs, Resend for email, and Razorpay for payments. Deployed as Docker Compose services on a single VPS for staging/production. Work proceeds in the same dependency order as `plan.md`'s own Task 1-6 breakdown, since each task's schema and services are genuine prerequisites for the next (infra before auth, auth/tenancy before course domain, course domain before both the learner journey and commerce, which can then proceed in parallel).

## Decisions & Rejected Alternatives

- **MVP boundary is exactly Tasks 1-6** — matches `plan.md`'s own "Recommended MVP boundary" section. Rejected: trimming further (payments/certs deferred) or expanding early (search/notifications pulled in) — the existing boundary was judged right-sized.
- **Public multi-tenant SaaS from MVP launch, not single-org-first** — any authenticated user can self-service-create an organization from day one; RLS-based isolation (already planned) makes this safe without additional schema work. Supersedes an earlier draft of Task 1 that framed the product as self-hosted software and `plan.md`'s "one organization per installation initially" phrasing. Rejected: gating org creation behind platform-owner approval, or deferring public signup to post-MVP — the platform owner explicitly wants public SaaS signup at launch.
- **License: Proprietary / All Rights Reserved** — since this is operated SaaS, not distributed software, customers never receive a copy of the code, so there's no OSS obligation. Rejected: MIT (no reason to open-source), AGPLv3 (copyleft protection is moot when nobody receives the code).
- **Platform monetization: commission on course sales, computed in Task 6** — not a separate organization subscription/billing system. Rejected: charging organizations a recurring platform fee (would need a whole new billing flow not built by any task); leaving monetization undefined for MVP (the product brief had drifted into implying org billing without any task building it).
- **Basic admin dashboard is built in Task 6**, not a separate task and not deferred — org-scoped (members, courses, enrollment/revenue) plus a platform-owner cross-org view, read-only, no bulk actions/feature flags/quotas. Rejected: a dedicated new task (adds a phase to the critical path for a small deliverable); dropping it from MVP entirely (Task 1's brief and `plan.md`'s own MVP boundary both call for a basic admin page, and it had no owning task before this fix).
- **Product name/domain deferred as a placeholder** — naming/branding (Task 1) must not block engineering start. Rejected: picking a name now — no name was ready and it isn't a technical dependency.
- **Go web framework: Gin** — widest adoption/middleware ecosystem/docs for the stack's chosen language. Rejected: Fiber (fasthttp, less net/http middleware compatibility), stdlib net/http + router (more plumbing to hand-roll).
- **Backend serves both server-rendered HTML and a versioned JSON API from day one** — avoids a rewrite when Task 12's mobile work starts. Rejected: HTML-only now, API designed later — would risk retrofitting auth/serialization concerns into existing handlers.
- **Supabase Auth is the identity source of truth**; Go verifies its JWTs and layers org/role/permission logic on top. Rejected: fully custom Go auth system — more control but re-implements security-critical, already-solved primitives (password hashing, verification, sessions).
- **Tenant isolation via Postgres RLS + `org_id` on every organization-owned table**, as defense-in-depth alongside Go-layer checks. Rejected: application-layer-only filtering — one missed `WHERE org_id = ...` becomes a silent cross-tenant leak with no DB backstop.
- **Go connects directly to Supabase Postgres (pgx) and issues `SET LOCAL` session variables per request/transaction** so RLS policies can read `current_setting()`. Rejected: routing all queries through Supabase's PostgREST/data API — too constraining for Go-side business logic and complex joins.
- **Migrations: golang-migrate** — plain SQL, framework-agnostic, works directly against the Supabase connection string. Rejected: Supabase CLI migrations (couples schema management to the Supabase CLI/dashboard), goose (equivalent but less-adopted alternative).
- **Payments: Razorpay** as the sole MVP provider, behind the adapter interface `plan.md` Task 6 already calls for. Rejected: Stripe as primary — Razorpay fits an India-focused market; Stripe can be added later as a second adapter.
- **Media storage split: bunny.net for video (streaming/CDN), Supabase Storage for everything else** (thumbnails, certificate PDFs, assignment submissions). Rejected: Supabase Storage for all media (no CDN/streaming optimization for video), bunny.net for all files (unnecessary for small non-video assets already covered by Storage + RLS).
- **Redis-backed job queue (e.g. asynq) for async work** — email sending, webhook processing, media callbacks — keeps request handlers fast and webhook retries reliable. Rejected: skipping Redis for MVP and handling these synchronously.
- **Email: Resend** for transactional email. Rejected: Postmark (more enterprise-oriented than needed yet), AWS SES (more setup overhead — DKIM, sandbox limits — before it can send freely).
- **Hosting: single VPS via Docker Compose** for staging and production. Rejected: managed container PaaS (Fly.io/Railway/Render) — higher cost/less control than needed pre-scale; deferring the decision to Task 12 — the plan needs a concrete target now to shape Task 2's deployment config.
- **Local dev uses the Supabase CLI local stack** (`supabase start`) for Postgres/Auth/Storage, not a shared cloud project or a bare Postgres container. Rejected: shared cloud dev project (data collisions between developers, needs network access), per-developer cloud project (more setup friction, still needs network access).
- **Local dev uses real bunny.net sandbox credentials for video**, not a stub. Rejected: mocking video upload/playback locally — risks hiding real integration bugs until staging.
- **Go app is one binary with `serve`/`worker` subcommands**, one Dockerfile/image, run as two Compose services via different entrypoint args. Rejected: separate binaries/Dockerfiles for API and worker — doubles build/CI surface for no MVP benefit.
- **Reverse proxy: nginx**, with TLS via certbot/Let's Encrypt (manual renewal setup). Rejected: Caddy's automatic HTTPS — user chose nginx despite the simpler auto-TLS tradeoff Caddy would have offered.
- **Task 2 builds a generic Redis-backed rate-limiting middleware; Task 3 applies it to auth routes.** Rejected: leaving all rate-limiting to Task 3 — would make Task 3 build limiter infrastructure from scratch instead of reusing Task 2's Redis.
- **No hard test-coverage percentage gate in CI for MVP** — coverage is measured/reported but doesn't block merges. Rejected: keeping an arbitrary 50% threshold — invites gaming with shallow tests or blocking legitimate PRs with no real baseline to justify the number.
- **Secret scanning: gitleaks** (official GitHub Action). Rejected: truffleHog — heavier, more configuration to avoid false positives for an MVP CI pipeline.
- **Redis persistence (AOF/RDB) enabled in production**, not just local dev. Rejected: treating production Redis as ephemeral/best-effort — Task 6's payment webhook jobs and email sends are queued here; losing them on a restart is a real problem.
- **Removed "feature flags" and "signing key rotation" mentions from Task 2** — neither corresponds to anything actually built in Tasks 1-6 (feature flags are `plan.md`'s post-MVP Task 10; there is no Go-side JWT signing key to rotate, since Supabase Auth signs JWTs and Go only verifies them). Rejected: keeping them as forward-looking placeholders — same pattern as Task 1's admin-dashboard drift, better to not promise what nothing builds.
- **Auth scope for MVP: email/password only**; Google OAuth and MFA deferred past MVP. Matches `plan.md`'s own sequencing ("OAuth only after password flow is stable") and MVP boundary ("Email authentication"). Rejected: including Google OAuth in MVP.
- **Block editor MVP scope: text, image, video, file, quiz blocks only** — matches the MVP boundary's explicit list ("Video, text, files, and quizzes"). Rejected: building the full block set (audio, embeds, callouts, code, downloads-as-distinct-type) up front — deferred to post-MVP incremental additions.
- **Certificate PDFs generated server-side in Go** (e.g. chromedp/HTML-to-PDF or a Go PDF library) rendering a branded template. Rejected: a third-party PDF/certificate API (DocRaptor, PDFShift) — adds a vendor and per-certificate cost not needed yet.
- **Tests written alongside MVP work for security/money-critical paths** — auth, tenant isolation, and payment/webhook logic get tests as Tasks 3 and 6 are built, rather than deferring all testing to Task 11 as the phase ordering might literally suggest. Broader test-suite breadth (accessibility, load tests) stays in Task 11.
- **Task 3 builds HTML+HTMX auth/account pages now, alongside the JSON API** — per the original spec, not deferred to a later frontend pass. Rejected: JSON-API-only for Task 3 with HTML added later — would have kept the task narrowly backend-focused but was overruled by the plan owner.
- **Email verification is mandatory before first login** — registration never returns a JWT; unverified accounts can't log in, create orgs, or accept invitations. Rejected: allowing login pre-verification with a nag banner — lower friction but lets unverified accounts create orgs/send invitations before confirming email ownership.
- **Account deletion is blocked while the user is a sole owner of any organization**, rather than cascading or auto-transferring. Rejected: cascade-deleting owned orgs (destroys other members' access/data without consent); auto-transferring to an implicit target (surprising, no clear rule for who's picked).
- **Orgs may have multiple simultaneous owners; ownership transfer reuses the existing role-change endpoint** (promote another member to `'owner'`, then optionally self-demote/leave) — no dedicated transfer-ownership endpoint. Rejected: a dedicated atomic transfer endpoint enforcing exactly one owner — safer against zero-owner states but adds surface area the spec didn't call for.
- **Failed-login backoff is two independent, bounded mechanisms**: the existing 5-attempts/15-min-per-IP limit, plus a per-email lockout (5 consecutive fails → 15 min lockout, doubling per repeat streak up to a 1-hour cap, reset on success). Rejected: a pure per-attempt exponential delay — closer to the spec's literal wording but harder to implement statelessly and test deterministically.
- **API tokens have optional expiry, default none (revoke-only)** — matches common API-token UX (e.g. GitHub PATs). Rejected: mandatory expiry with a max lifetime — more secure by default but needs rotation-reminder UX out of scope for MVP.
- **Invitations to an email with no existing account route through registration, then auto-accept** — accept-endpoint returns `registration_required`, the invited person registers/verifies, and the pending invitation is applied automatically. Rejected: requiring a pre-existing account before accepting — simpler server logic but a worse, disconnected two-step UX.
- **Task 3 sends a plain transactional invitation email now**, via Task 2's Resend integration — invitations must work end-to-end in this task. Rejected: stubbing/logging the invite link and leaving delivery to Task 7 — would leave invitations non-functional until Task 7 ships.
- **`'moderator'` carries no elevated permissions in Task 3** — it's a valid role value but equivalent to `'learner'` in this task's permission matrix; only `'owner'` has elevated rights. Task 7 (Communities) defines moderator's real discussion-moderation powers later. Rejected: giving moderators member-management rights now — front-loads a permission design Task 7 might want differently.
- **No hard cap on organizations a user may create/own for MVP**, just rate-limiting the create-org endpoint (reusing Task 2's rate limiter, e.g. 5/hour/user). Rejected: a fixed max-owned-orgs cap — simpler abuse prevention but an arbitrary number needing a support-ticket escape hatch.
- **HTML pages use stdlib `html/template`**, no third-party templating library. Rejected: templ (a-h/templ) — more ergonomic component composition but adds a compiled-template build step and a dependency the original stack never named.
- **CSRF middleware protects all state-changing HTML/HTMX form routes**; JSON bearer-token API routes are exempt. Rejected: relying on SameSite cookies alone — weaker defense-in-depth once cookies drive real mutations.
- **HTML session cookie holds an opaque session ID only, resolved server-side via Redis to the stored Supabase JWT/refresh-token pair** — never the JWT itself. Rejected: putting the JWT directly in an HttpOnly cookie — simpler (no session store) but logout can't truly revoke before natural expiry, and a stolen cookie is a stolen JWT.
- **URL slug always wins for org-context resolution** when a route has `:org_slug` — the caller's role is re-resolved fresh from `memberships` for that org, 403ing non-members; session/header context is a fallback only for org-less routes. Rejected: letting the session's stored "current org" always govern — risks silently acting on a stale org when the URL says otherwise.
- **First platform owner is set via a one-time `serve bootstrap-owner` CLI subcommand**, not an env var. Rejected: env-var-driven auto-promotion on every boot — zero manual step, but leaves a standing config-driven privilege-escalation path.
- **No app-side password policy; deferred entirely to Supabase Auth's defaults.** Rejected: enforcing a stricter Go-side policy — more UX control but duplicates/risks drifting from validation Supabase already does.
- **Task 4 extends Task 3's `permissionMatrix`/`Can(role, action)` pattern** for course-domain authoring checks, using the actual role constant `'owner'` (correcting the spec's earlier, inconsistent `'org_owner'` naming). Rejected: ad-hoc inline role checks per handler — would drift from Task 3's established, auditable permission model.
- **`sort_order` on chapters/lessons/blocks is `NUMERIC(20,10)` with self-healing renormalization** — the reorder endpoint renumbers siblings to whole-number spacing in-transaction when fractional precision runs low, instead of ever needing a client-side full-list reorder. Rejected: float8 (precision loss risk) or plain integer-with-gaps (touches more rows per reorder on average).
- **Video goes through Bunny Stream (not a plain Storage Zone), with one Stream library per organization for hard tenant isolation**, provisioned lazily on an org's first video upload (adds a nullable `bunny_library_id` to `organizations` via a Task 4 migration, rather than reopening Task 3). Transcode completion is delivered via Bunny's webhook, HMAC-verified and processed through Task 2's asynq worker — never trusted from an unverified call, same rule already applied to payment webhooks. Rejected: shared library + polling (weaker isolation, added polling load); plain Storage Zone + custom ffmpeg transcoding (more infrastructure to own, no vendor-managed pipeline).
- **Chapter/lesson delete rejects with 409 if non-empty** rather than cascading. Rejected: cascade soft-delete (extra bookkeeping columns/queries for entities that don't otherwise need them) or cascade hard-delete (irreversible, high accidental-data-loss risk).
- **Course hard-delete is allowed only when `status = draft`**, with no runtime "attempts exist" check in Task 4 since the `attempts` table doesn't exist until Task 5 — a draft course can never have attempts under any task's design. Task 5's migration adds a foreign key from `attempts` to `courses` so the database itself keeps this safe going forward. Rejected: blocking all hard-delete in Task 4 — leaves an explicit acceptance criterion unmet.
- **Text-block HTML sanitized server-side via `bluemonday`** with a fixed allowlist (`p, strong, em, u, ul, ol, li, br, a[href], h1-h3`). Rejected: regex/manual tag stripping — a well-known source of XSS bypass bugs.
- **Quiz authoring supports `mcq`, `true_false`, and `short_answer`** — `short_answer` stores `accepted_answers: string[]` (not a single `correct_answer_text`) so Task 5 can support multiple accepted phrasings without a later migration. Rejected: `mcq`/`true_false` only for Task 4, deferring `short_answer`'s shape entirely to Task 5 — would risk a schema migration when Task 5 lands.
- **Categories are a curated, owner-only-managed taxonomy (explicit CRUD); tags are freeform get-or-create** on course-tagging, available to teacher or owner. Rejected: full CRUD for both, teacher+owner — would let categories sprawl as freely as tags, undermining the taxonomy's usefulness for browsing.
- **No asset storage cleanup/hard-delete in Task 4** — orphaned assets (referenced by zero blocks, e.g. after course duplication diverges) are left in place; a future task can sweep truly-unreferenced ones. Rejected: reference-counted hard-delete — correct but adds real complexity (concurrent decrement races, partial-failure cleanup) to an already-large task.
- **Video assets carry a `processing_status` (pending/processing/ready/failed)**; the editor polls via HTMX until ready, and publish rejects any course containing a non-ready video block. Rejected: treating an asset as ready immediately with no status tracking — risks a publish succeeding with a broken video and no signal to the teacher.
- **Task 4 builds its own minimal read-only preview template** rather than depending on Task 5's not-yet-built learner player. Rejected: deferring `/preview` to Task 5 — would drop an explicit Task 4 acceptance criterion and block on a task that hasn't started.
- **Fixed a lifecycle-ordering bug in the original Task 4 draft**: `scheduled` is now reachable from `review` (setting a future `publish_date`, before first publish) rather than from `published` as originally (incorrectly) written; a scheduled asynq job auto-transitions `scheduled` → `published` when the date arrives, taking the version snapshot at that moment. Rejected: implementing the spec literally — `published` → `scheduled` serves no real product need (already-visible courses don't need a future publish date).
- **Every publish unconditionally creates a new `course_versions` snapshot**, even if content is unchanged since the last one. Rejected: diffing to skip redundant snapshots — not worth the added complexity for MVP; a redundant JSONB row is cheap.
- **Non-video upload completion requires backend HEAD verification against Supabase Storage**, not just a client-reported "done" call, before the `assets` row is created. Rejected: trusting client-reported completion alone — lets a client forge asset metadata for a file that was never actually uploaded.
- **Task 4 writes RLS-isolation, permission-matrix, status-transition, and signed-URL-scoping tests alongside the task** — continuing Task 3's precedent that security-critical paths aren't deferred to Task 11. Rejected: unit-testing only pure logic (sort_order, duplication, transitions) and deferring isolation/permission tests — a cross-tenant RLS bug is the highest-risk failure mode this task can introduce.
